What is Property Wrappers.?

A Property Wrapper is a feature that enables you to add custom behavior to properties. It allows you to define a reusable logic for property access and modification, making your code cleaner and more maintainable.

Usage:

A property wrapper can be used with structs, classes, and enums. The init() of the property wrapper is inherently private, so if we want to make use of it, we must explicitly add init() to the property wrapper struct, ensuring that the wrapped value is mandatory and defined within the struct.

@State

  • @State property wrapper allows us to modify values inside a struct, which would normally not be allowed because structs are value types.
  • When we use @State before a property in SwiftUI, we move its storage to a special place managed by SwiftUI. This allows SwiftUI to recreate our structure whenever necessary without losing the data it was holding.
  • If we donâ€™t declare @State property, it means we can not change the value of that property.
struct StateExample: View {
  @State private var intValue = 0
  @State private var modelObject = MyModelObject()

  var body: some View {
    VStack {
      Text("intValue equals \(intValue)")

      Button("Increment") {
        intValue += 1
      }
    }
  }
}
  • Now, should we use state property for all UIControls?
  • Yes. In the majority of cases in SwiftUI, you will be using state property for your control. This helps you to update the state of your UI control. You donâ€™t need any delegate or data source methods, and with a few lines of code, we can make those reactive changes immediately.

You should use @State if:

  • The view itself creates (and owns) the instance you want to wrap.
  • You need to respond to changes that occur within the wrapped property.
  • Youâ€™re wrapping a value type (struct or enum).
  • iOS 17+: Youâ€™re wrapping an @Observable object.

@Binding

One of the things that SwiftUI promotes is building reusable components in your app. There can be multiple reusable components in one view. When you want the child viewâ€™s changes in the parent view, you can use a binding property wrapper. By using this in the child view, you can read or modify the parent viewâ€™s source of truth.

  • @Binding does not accept any default value.
  • @Binding has to be public, not private.
  • You have to use $ to pass a binding reference because, without $, Swift will pass a copy of the value instead of passing a bindable reference.
struct BindingView: View {
  @Binding var intValue: Int

  var body: some View {
    VStack {
      Button("Increment") {
        intValue += 1
      }
    }
  }
}

struct StateView: View {
  @State private var intValue = 0

  var body: some View {
    VStack {
      Text("intValue equals \(intValue)")

      BindingView(intValue: $intValue)
    }
  }
}

You should use @Binding if:

  • You need read- and write access to a property thatâ€™s owned by a parent view.
  • The wrapped property is a value type (struct or enum). (You can also use an @Binding for reference types (class) but it's not nearly as common.)
  • You donâ€™t own the wrapped property (itâ€™s provided by a parent view).

  • @Published is the most useful property wrapper in SwiftUI. It allows the observable objects that automatically announce when changes occur.
  • SwiftUI automatically observes changes and re-invokes the body property of views that rely on the data. Whenever an object property marked @Published is changed, all views using that object will be reloaded to reflect those changes.

@StateObject

@StateObject is similar to @State property, except itâ€™s applied to ObservableObjects. An ObservableObject protocol is always a reference type (class) and informs SwiftUI whenever one of its @Published properties changes.

  • When you need to create a reference type inside one of your views and ensure it stays alive for use in that view and others you share it with, thatâ€™s where @StateObject is used.
  • A property marked as @StateObject will keep its initially assigned instance as long as the view is needed, even when the struct gets recreated by SwiftUI.
  • This is the same behavior you see in @State, except itâ€™s applied to an ObservableObject rather than a value type like a struct.

ObservableObject is a class protocol because it is inherited from anyObject.

  var body: some View {

You should use @StateObject if:

  • You want to respond to changes or updates in an ObservableObject.
  • The view youâ€™re using @StateObject in creates the instance of the ObservableObject itself.

There are three key things we need to work with:

  • Any type marked with @ObservedObject must conform to the ObservableObject protocol.
  • Observed objects are specifically designed for data that is external to your view, which means it might be shared across more than one view. The @ObservedObject property wrapper will automatically make sure the property is watched closely so that important changes will reload any views using it. This also means the data must be created elsewhere, and then sent into your view.
  • Not all properties in an observed object cause views to refresh â€“The @Published property wrapper is added to any properties inside an observed object that should cause views to update when they change.
struct DataOwnerView: View {
  @StateObject private var provider = DataProvider()

  var body: some View {
    VStack {
      Text("provider value: \(provider.currentValue)")

      DataUserView(provider: provider)
    }
  }
}

struct DataUserView: View {
  @ObservedObject var provider: DataProvider

  var body: some View {
    // create body and use / modify `provider`
  }
}

ObservedObjects are usually used when the data is stored outside of the view, so when the view is redrawn, you wonâ€™t lose the data. StateObjects are usually used when that data needs to be stored inside the view.

You should use @ObservedObject if:

  • You want to respond to changes or updates in an ObservedObject.
  • The view does not create the instance of the ObservedObject itself. (if it does, you need a @StateObject)

@EnvironmentObject

SwiftUIâ€™s @EnvironmentObject property wrappers allow us to create views that rely on shared data, often across an entire SwiftUI app.

  • @EnvironmentObject is created with and stored at the application level. Itâ€™s shared data that every view can read/write if they want to.
  • Since all views point to the same model, if one view changes the model, all views immediately update, i.e., thereâ€™s no risk of getting different parts of your app out of sync.

You should use @EnvironmentObject if:

  • You would normally use an @ObservedObject but you would have to pass the ObservableObject through several view's initializers before it reaches the view where it's needed.

@Environment

@Environment property observer is used when we might need to read a device setting or device environment to adjust our view or behavior to match user needs. Examples of these values are current device locale, device current calendar, color scheme (dark or light), and size class.

struct MyView: App {
  @Environment(\.colorScheme) var colorScheme: ColorScheme

  var body: some View {
    Text("The color scheme is \(colorScheme == .dark ? "dark" : "light")")
  }
}

You should use @Environment if:

  • You want to inject some value into the SwiftUI environment using a key.
  • The injected property doesnâ€™t need to behave like an @EnvironmentObject.
  • The property should be available to all views that are subviews of the view that received the .environment modifier.

@AppStorage

The @AppStorage property wrapper is an app-wide wrapper around UserDefaults. This means that it's great for storing simple key / value pairs. When the data in UserDefaults changes, your view reloads. You can update values in @AppStorage by assigning a value to your property.

struct ContentView: View {
    @AppStorage("lastTap") var lastTap: Double?

    var dateString: String {
        if let timestamp = lastTap {
            return Date(timeIntervalSince1970: timestamp).formatted()
        } else {
            return "Never"
        }
    }

    var body: some View {
        Text("Button was last clicked on \(dateString)")

        Button("Click me") {
            lastTap = Date().timeIntervalSince1970
        }
    }
}

You should make use of @AppStorage the when:

  • Youâ€™re storing simple user preferences.
  • You want to track simple data like when the user last launched your app.
  • You need to persist some very simple state that should service an app restart.

@SceneStorage

The @SceneStorage property wrapper is similar to @AppStorage except it only persists data local to the scene that your view is currently in. On iOS an app will typically have a single scene but on the Mac and iPad, an app can have several scenes. When a scene is temporarily torn down and restored later, your scene storage will be available again. If a scene is destroyed, all scene-related data is destroyed too.

You should only use scene storage for state-related data that are non-essential to your app.

struct ContentView: View {
    @SceneStorage("lastTap") var lastTap: Double?

    var dateString: String {
        if let timestamp = lastTap {
            return Date(timeIntervalSince1970: timestamp).formatted()
        } else {
            return "Never"
        }
    }

    var body: some View {
        Text("Button was last clicked on \(dateString)")

        Button("Click me") {
            lastTap = Date().timeIntervalSince1970
        }
    }
}

You should make use of @SceneStorage when:

  • Youâ€™re storing a simple state related to the current scene
  • The data youâ€™re persisting in isnâ€™t sensitive or mission-critical.
